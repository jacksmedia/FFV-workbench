# String Table Amendments for ff5u-sfc.json

## Summary

This PR adds **3,803 named entries** across four string tables in `ff5u-sfc.json`,
replacing the previous 325 entries. Every reference to events, triggers, NPC
switches, and event switches throughout the webapp now shows a meaningful label
instead of a generic "Event 742" or "NPC Switch 591".

| String Table | Before | After | Coverage |
|---|---|---|---|
| Event Switches | 8 | 464 | 90% of 512 |
| NPC Switches | 298 | 746 | 80% of 922 |
| Event Scripts | 21 | 1,914 | 98% of 1,940 |
| Trigger Scripts | 3 | 679 | 96% of 704 |
| **Total** | **325** | **3,803** | |

All 325 original human-authored names are preserved exactly as-is. The new
entries sit alongside them, filling in the previously unnamed majority.

**All 3,803 names are 100% unique** within each string table -- no duplicates.

No JavaScript changes are required. This is purely a string table expansion.

---

## What's Not Changed

- All 325 original human-authored names are preserved verbatim
- No structural changes to the JSON (no new keys, no schema changes)
- No changes to assembly definitions, hierarchy, encoding, or any other section
- No JavaScript changes required

The only modification is the `"string"` object within four existing
`stringTable` blocks, expanding them with additional key-value pairs.



## What Changed

Labels are generated from the ROM's own data -- event bytecode, dialog text,
trigger definitions, NPC tables, map structures, monster names, and battle
formations -- then enriched with inline cross-references. Opaque numeric
IDs are resolved to descriptive names wherever possible:

| Reference | Resolved To | Example |
|---|---|---|
| `battle(264)` | Monster name | `Sekmet(Bat.264)` |
| `call(1407)` | Event name | `call:Restore HP/MP(Ev.1407)` |
| `Shop #12` | Shop name | `Karnak - Equipment(Sh.12)` |
| `event_switch` | Switch purpose | `ESw.269:Obtained Cabin!=ON` |
| `npc_switch` | NPC role | `NSw.544:It's locked.=OFF` |

Where duplicate names occur (e.g., two events that both do
`[obj_action, sound]`), they are disambiguated with a `(n)` prefix:
`(1) obj_action, sound`, `(2) obj_action, sound`.


### Event Switches (8 -> 464)

Event switches are single-bit flags toggled by event scripts and checked by
triggers and NPCs. The original definition named 8 of 512 (notable ones like
"Obtained Cornago Jar" and "Disable Random Battles"). The remaining 504 showed
as "Event Switch N".

New names were generated by tracing cross-references across four data sources:

- **Setter events**: Which events turn the switch ON or OFF, and what dialog
  they contain (e.g., `Surgate Castle Library - "SECRET OF LONKA"`)
- **Map placements**: Where those setter events are placed on the world map,
  resolved through event triggers, trigger script placements, and NPC caller
  chains
- **NPC gates**: Which NPCs are shown/hidden based on the switch
  (e.g., `Tule - Pub Dancer (gated)`)
- **Trigger/NPC checks**: Where the switch is read, used as a fallback when
  setter data is unavailable

Each name includes the map location where the switch is set and a dialog
snippet or event description. An enrichment pass then resolves any opaque
`battle()` or `call()` references within the name to actual monster/event
names.

| Tier | Count | Example |
|---|---|---|
| Original (human-authored) | 8 | `Obtained Cornago Jar` |
| Global flag (dialog only) | 8 | `"Can't carry any more items!"` |
| Setter dialog + map | 280 | `Castle Tycoon Storehouse - "<Bartz>: It's empty!"` |
| Setter event + map (enriched) | 62 | `Karnak Castle, Sekmet(Bat.264), ESw.49:Obtained Elixir!=ON` |
| NPC gated | 40 | `Tule - Pub Dancer (gated)` |
| Map only (progress flag) | 51 | `Istory - progress flag` |
| Setter only (no map) | 9 | `Ev.171: Ronka Ruins Flying Fortress #2` |
| Check/clear only | 6 | `cleared by event 323` |

**Verification**: An automated audit confirmed that 100% of map-bearing names
(396/396) have their named map present in at least one cross-reference source
(setter event placement or checker context). Global flags (switches set across
6+ different maps) use dialog-only names to avoid misleading single-map
attributions. Subroutine events (placed on 4+ maps, like Inn or Piano scripts)
are detected and excluded from map resolution.

### NPC Switches (298 -> 746)

NPC switches control NPC visibility and behavior per map. The original 298
entries were already well-named. The new 448 entries follow the same convention,
generated from event cross-references and map placements. Bracket-desc names
have their opaque references enriched inline.

| Tier | Count | Example |
|---|---|---|
| Original (human-authored) | 298 | `Pyramid of Moore - First Tablet` |
| Map + dialog | 275 | `Wind Shrine 1F - "The Wind Crystal was being controll..."` |
| Map + event description (enriched) | 99 | `Fire-Powered Ship Bulkhead, call:Restore HP/MP(Ev.1407), sound` |
| Map only | 62 | `Moogle Village - NPC` |
| Event/dialog only | 12 | `Ev.323: obj_action, NSw.8=ON` |

### Event Scripts (21 -> 1,914)

Event scripts are the game's primary scripting system -- dialog, cutscenes,
battles, item grants, and world changes. The original definition named 21
(including "Game Start", "Intro", "Ending", and "Piano Practice 1-8"). The
new entries name 1,914 of 1,940 scripts.

Names are derived from the script's own content, then enriched with map
context and resolved cross-references:

- **Dialog-based** (579 scripts): The opening dialog line becomes the name,
  e.g., `"Galuf: This is it, the Big Bridge!"`
- **Call with dialog** (30 scripts): Subroutine calls that contain dialog,
  annotated with event ID, e.g., `call(Ev.44): "Are you okay? Lenna: Y..."`
- **Shop/Inn** (109 scripts): Named by function with shop name resolved,
  e.g., `Karnak - Equipment(Sh.12)` or `Inn (100gp)`
- **Map + enriched action** (414 scripts): A structural summary of the
  script's opcodes with all references resolved, prefixed with the map
  where the event runs, e.g.,
  `Karnak Castle, Sekmet(Bat.264), ESw.49:Obtained Elixir!=ON` or
  `Castle Exdeath 1F, call:Restore HP/MP(Ev.1407), sound`
- **Map transition** (271 scripts): Events that transfer the player to
  another map, e.g., `World Map (Bartz' World) #1`
- **Enriched action only** (485 scripts): Structural summary without map
  context (event not placed on any specific map or placed on too many),
  e.g., `call:Inn party heal(Ev.42), obj_action`
- **Empty/unused** (26 scripts): Marked as `[empty/unused]` (excluded from
  the string table)

### Trigger Scripts (3 -> 679)

Trigger scripts fire when the player steps on a tile or presses a button.
The original definition named 3 ("Tent", "Cabin", "Save Point"). The new
entries name 679 of 704 scripts.

Names combine the trigger's map placement with enriched event references:

| Pattern | Example |
|---|---|
| Map + dialog | `Crescent, "He.. here it comes...! Another"` |
| Map + enriched event | `World Map (Bartz' World), Gohn Town of Ruin #1(Ev.208)` |
| Map + switch condition | `Castle Walse 3F, if Quelb Gate is Locked OFF(ESw.100)` |
| Map + button press | `Surgate Castle Library, "It's the JKL bookshelf." (A)` |

The `(A)` suffix indicates button-press triggers (player must press A to
activate), distinguishing them from step-on triggers. Event names used in
trigger labels have their map prefix stripped when it would duplicate the
trigger's own map context.

## How It Was Done

Names were generated by a custom Python pipeline that cross-references the
game's ROM data:

1. **Data extraction**: Crawler scripts parsed the ROM's event bytecode,
   trigger definitions, NPC tables, and map structures into structured JSON
   files (event names, trigger scripts, NPC switches, event switches,
   entrance triggers, and a map atlas).

2. **Cross-reference resolution**: A naming script built indices mapping
   events to maps (via event trigger placements, trigger script placements,
   and NPC caller chains), then traced which events set each switch and where
   those events run.

3. **Name generation**: Each resource type has its own naming logic that
   combines map context with dialog snippets or structural descriptions,
   using a tiered priority system that prefers more specific/informative
   names.

4. **Quality pass**: A post-processing script enriches event script names
   with map context (420 events gained map prefixes) and resolves `call()`
   targets to human-readable names where the target is a well-known event.

5. **Label enrichment**: A comprehensive enrichment script resolves all
   opaque numeric references to descriptive inline annotations:
   - **Monster names** extracted from ROM ($E00050, 384 entries with DTE
     decoding) and mapped through battle formation data ($D03000, 512
     formations)
   - **Call targets** resolved to their event script names
   - **Shop IDs** resolved to shop names from the existing string table
   - **Event/NPC switches** annotated inline with their purpose and state
   - Pre-enrichment switch names are used for compact inline annotations,
     with circular-reference detection to prevent self-referential labels

6. **Deduplication**: Any remaining duplicate names within each table
   receive a `(n)` prefix (e.g., `(1) obj_action, sound`,
   `(2) obj_action, sound`), ensuring 100% uniqueness.

7. **Injection**: A merge script rebuilds each string table in
   `ff5u-sfc.json`, always preserving the original human-authored entries.

8. **Verification**: An automated audit cross-validates every generated name
   against all known map contexts, flagging any discrepancies. The current
   output passes with zero mismatches.

The pipeline is idempotent -- running it again produces the same output. If
the original definition is updated with new human-authored names, they will
take priority over auto-generated ones on the next run.
