Notes supporting the hunt for the OAM palette reference:

Carefully read all six screenshots. Here's the full breakdown.

  A9 14 not found in any of these regions. The DMA source for the KO tiles probably gets loaded as a full 16-bit or
  24-bit immediate in a different path, or inside a generic VRAM-transfer utility rather than inline. Let's set that
  search pattern aside for now — there's a better lead.

  ---
  Strongest candidate: Screenshot 2, starting at $017D8C

  Reading the bytes carefully across the row boundary:

  017D8C: A9 E0        LDA #$E0
  017D8E: 8D 88 BC     STA $BC88
  017D91: 8D 89 BC     STA $BC89
  017D94: 8D 8A BC     STA $BC8A
  017D97: 8D 8B BC     STA $BC8B
  017D9A: A9 00        LDA #$00
  017D9C: 8D 84 D1     STA $D184
  017D9F: 9C F0 D0     STZ $D0F0
  017DA2: 9C 85 BC     STZ $BC85
  017DA5: 60           RTS

  That's a standalone function. $E0 = 1110 0000 — in OAM attribute byte format that's: Y-flip set, X-flip set, priority
  2, palette 0. It writes the same hardcoded attribute to all four party slots at $BC88-$BC8B and returns. This is
  almost certainly the OAM attribute initializer that locks all character sprites to palette 0 — including the KO pose.

  What immediately precedes this at $017D78 is the 6-entry jump table for the animation state machine ($D102 AND $07
  dispatch), with handler addresses $7E1F, $7E48, $7E6B, $7E1B, $7E44, $7E8B (all bank C1). This function at $017D8C
  sits right after that table — it's likely called to reset/initialize all four slot attributes before the individual
  per-character handlers run.

 What the memory viewer confirms ($017D80–$017DE0):

  Reading carefully across row boundaries, I can now decode two distinct functions:

  Function A — at $017D8C (right after the 6-entry jump table):
  017D8C: A9 E0       LDA #$E0       ; palette 0, priority 2, flip bits
  017D8E: 8D 88 BC    STA $BC88      ; → all 3 char OAM attributes get $E0
  017D91: 8D 89 BC    STA $BC89
  017D94: 8D 8A BC    STA $BC8A
  017D97: 9C 02 D1    STZ $D102      ; zero animation state counters
  017D9A: 9C 03 D1    STZ $D103
  017D9D: 9C 04 D1    STZ $D104
  017DA0: 9C F0 D0    STZ $D0F0
  017DA3: 9C 85 BC    STZ $BC85      ; ← clears the flag seen in debugger
  017DA6: 60          RTS

  Function B — at $017DC0 (a second gated write):
  017DC0: A9 02       LDA #$02
  017DC2: 85 88       STA $88
  017DC4: AD 04 D1    LDA $D104
  017DC7: 25 88       AND $88        ; test bit 1 of $D104
  017DC9: F0 0E       BEQ → $017DD9  ; if clear → branch
  017DCB: A9 E0       LDA #$E0       ; same palette 0 hardcode
  017DCD: 8D 88 BC    STA $BC88
  017DD0: 8D 89 BC    STA $BC89
  017DD3: 8D 8A BC    STA $BC8A
  017DD6: 4C A7 7D    JMP $7DA7      ; return path
  017DD9: A9 1F       LDA #$1F       ; ← branch target when bit clear
  017DDB: 8D 11 D1    STA $D111      ; store $1F into state registers
  017DDE: 8D 12 D1    STA $D112

  The $1F stored to $D111/$D112 is the exact value the debugger code checks!

  ---
  What the debugger shows:

  C1125C: LDA $04F2 [$7E04F2] = $17   ; current sprite-state for this character
  C1125F: CMP #$1F                     ; is it the KO-lying-down state?
  C11261: D0 2F  BNE $C11292           ; $17 ≠ $1F → branch taken (no KO right now)
  C11263: LDA $BC85 [$7EBC85] = $00    ; (only reached when state IS $1F)
  C11266: D0 2A  BNE $C11292
  C11268: LDA $BC9B [$7EBC9B] = $00
  C1126B: ORA #$20
  C1126D: STA COLDATA                  ; sets SNES color math for KO sprite tinting

  $04F2 is the sprite-state field for the character currently being processed in a per-character render loop. $1F =
  state 31 = KO lying-down pose. The game is running a normal battle right now ($04F2 = $17 = some alive-state), so the
  BNE fires and the KO path is skipped entirely. That's why $017D8E never showed up in your PC trace — you need a KO'd
  hero for this code to fall through.

  The $BC85 check gates the COLDATA (color-tinting) effect, and $BC85 is zeroed by Function A above when the animation
  resets.

  ---
  The core picture:

  Both Function A and Function B write the hardcoded $E0 (palette 0) to $BC88–$BC8A. These are the OAM attribute bytes
  for the 3 visible character slots. The path to them is: animation state becomes $1F → $D111/$D112 get set to $1F →
  eventually Function B fires (gated on $D104 bit 1) and locks in palette 0.

  ---
  How to confirm cleanly without the full debugger:

  1. Get into a battle and let Bartz get KO'd
  2. Pause the emulator
  3. In the Memory Viewer, switch to WRAM (not PRG ROM) and check address $04F2
  4. If it reads $1F, the KO state is active and that's the value gating everything

  Once you confirm $04F2 = $1F during KO, the patch site is $017DCB — replace A9 E0 (LDA #$E0, hardcoded palette 0) with
   a JML to freespace that reads the character's party slot from the same context and substitutes the correct palette
  value into the attribute byte before writing to $BC88–$BC8A.
